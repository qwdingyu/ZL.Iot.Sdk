# Source-Verified Design Document Writing

## When to Use

当用户要求为某个项目模块/组件编写设计文档（架构、运行流程、说明文档等），且要求"认真仔细"、"不要出现任何幻觉"、"采用行业最佳实践"时，使用此技能。

## Core Principle

**所有文档断言必须经过源码逐文件验证，不基于假设或推断。无法确认的内容明确标注"未验证"或从文档中排除。**

## Procedure

### Phase 1: 全量源码收集

1. **列出模块文件结构** — `list_directory` 逐层级遍历，获取完整文件树
2. **逐文件读取** — 使用 `read_file` 读取每个 .cs/.ts/.go/.py 等源文件全文
3. **读取项目文件** — .csproj/.tsconfig/go.mod/package.json 等构建配置
4. **读取解决方案/工作区** — .sln/workspace 等，确认模块是否在构建体系中
5. **读取文档文件** — README.md、现有设计文档等

### Phase 2: 断言验证

对初始设计文档草稿中的每个断言进行验证：

1. **类/接口/字段是否存在** — `read_file` + grep 确认
2. **公开 API 签名** — 逐文件确认方法签名、参数、返回值
3. **内部实现细节** — 确认关键方法（循环、条件分支、异常处理）的真实逻辑
4. **配置项是否被实际使用** — grep 配置属性名，确认代码中是否有读取（不仅是赋值/定义）
5. **方法是否被实际调用** — grep 方法名，确认调用链存在
6. **依赖是否实际使用** — grep 命名空间/类型引用，确认真正 import 了哪些
7. **解决方案包含性** — grep `.sln` 确认模块是否在项目解决方案中
8. **编译验证** — `dotnet build` 等确认源码可编译通过

### Phase 3: 死代码审计

主动搜索以下模式：

| 模式 | 搜索策略 |
|------|----------|
| 配置属性被定义但从未读取 | grep 配置属性名 → 在业务代码中搜索引用 |
| 方法被定义但从未调用 | grep 方法名 → 确认调用链 |
| 字段被赋值但从未读取 | grep 字段名 → 确认使用点 |
| using 引用但类型未使用 | grep using 行 → 搜索对应命名空间中的类型 |
| NuGet 包引用但类型未使用 | grep PackageReference → 搜索对应命名空间的引用 |
| 示例代码引用不存在的类型 | 对比 README 中的示例代码与实际源码文件列表 |

### Phase 4: 安全隐患审计

搜索以下模式：

| 风险类型 | 搜索模式 |
|----------|----------|
| SQL 注入 | 字符串拼接表名/列名到 SQL |
| 路径注入 | 用户输入拼接到文件系统操作 |
| 连接泄漏 | 未 using/Dispose 的连接或连接池 |
| 静态资源无生命周期 | static 字段永不回收 |
| 敏感信息明文 | 配置文件中连接字符串/密钥明文 |

### Phase 5: 文档重写与验证

1. **基于验证结果重写文档** — 删除所有未验证的断言
2. **标注每个关键决策** — 标注"经源码确认"或"基于 README 描述"
3. **区分事实与推测** — 无法确认的功能标注为"预留/未实现"
4. **附录包含审计发现** — 死代码、未使用依赖、安全隐患独立章节
5. **文档末尾附验证声明** — 声明哪些经过源码验证，哪些未验证

## Verification Checklist

在文档输出前，逐项确认：

- [ ] 所有类名、方法名、字段名与源码一致
- [ ] 所有 API 签名（参数类型、返回值）与源码一致
- [ ] 所有"死代码"断言已通过 grep 搜索确认未使用
- [ ] 所有"未使用依赖"断言已通过 grep 搜索命名空间确认未引用
- [ ] 所有"安全隐患"断言已定位到具体代码行
- [ ] 所有架构描述与源码中的文件结构和命名空间一致
- [ ] 解决方案 (.sln) 包含性已验证
- [ ] 编译状态已验证（通过或不通过）
- [ ] 没有将 README.md 中的示例代码/描述当作源码实际功能
- [ ] 文档末尾附有验证声明

## Anti-Hallucination Rules

1. **永不基于"看起来应该这样"写断言** — 必须 grep/read_file 确认
2. **README.md != 源码事实** — README 可能有过期示例或不存在的类型引用
3. **类注释 ≠ 实际行为** — 类注释可能描述未来计划，不是当前代码
4. **字段赋值 ≠ 字段使用** — 赋值只是第一步，必须确认有读取点
5. **方法定义 ≠ 方法调用** — 定义存在不代表被调用链引用
6. **命名空间引用 ≠ 实际使用** — using 声明了不代表代码中用到了该命名空间的类型
7. **配置存在 ≠ 功能生效** — 配置项定义不代表业务逻辑中读取了它
8. **设计文档 ≠ 源码事实** — 设计文档可能描述计划中的功能或错误的分析，**编写设计文档或实现集成时，必须读取集成方的实际源码验证，而非基于另一方或设计文档的推断**

### Critical Anti-Hallucination Pattern: Cross-System Integration

**当涉及跨系统集成时，必须读取对方系统的实际源码确认以下维度，不可基于设计文档或自己的推测**：

| 必须验证的维度 | 示例（ZL.DataSync × PcStationIot 教训） |
|----------------|------------------------------------------|
| 增量同步条件 | 文档说 `_UploadFlag`，实际源码用的是 `ProcessTime > @lastTime` |
| 水位线表结构 | 文档写复杂结构，实际是 `TableName (PK), SyncTime (TEXT)` |
| 写入后标记逻辑 | 文档暗示需要标记业务表，实际只更新水位线表不碰业务表 |
| 配置项存在性 | 设计文档说支持某配置项，实际配置类中可能还没有 |

**经验教训**: 在一次真实的跨系统集成任务中，基于设计文档的第10节分析产生了3个幻觉：
- 幻觉1：增量条件使用了 `_UploadFlag > @lastTime`（实际是 `ProcessTime > @lastTime`）
- 幻觉2：_SyncLog 表结构被错误描述为复杂多字段表（实际只有 TableName + SyncTime）
- 幻觉3：在业务表中标记 `_UploadFlag`（实际不修改业务表）

**修正方法**: 直接读取 PcStationIot 的 `RemoteSyncService.cs` 源码，确认实际实现后完全重写策略逻辑。

## Output Format

设计文档应包含以下章节：

1. 概述（特性、约束）
2. 设计动机与背景
3. 架构总览（文件结构、分层图、并发模型）
4. 核心组件设计（每个组件的职责、内部字段、公开 API、关键实现细节）
5. 运行流程（生命周期、关键路径、流程图）
6. 配置设计（完整字段表、注册方式、示例、校验规则）
7. 数据流与同步语义
8. 扩展点设计
9. 注意事项与已知问题（必须注意 + 代码层问题 + 安全隐患）
10. 配置复杂度评估
11. 故障排查指南
12. 测试策略
13. **源码审计：死代码、未使用依赖、安全隐患**（独立章节）
14. 验证声明
