# Cross-System Integration Analysis

## When to Use

当需要评估两个独立开发的系统/模块之间的集成可行性，且双方已有各自的数据同步方案时，使用此技能。典型场景：
- 新引入一个数据同步库，需要评估是否能替换现有自定义同步逻辑
- 合并两个系统的通信协议
- 评估第三方库与现有代码的兼容性

## Core Principle

**所有差距断言必须经过双方系统的源码逐文件验证，不基于 README 描述、类名暗示或"看起来应该这样"的推断。对比表中的每一行都必须是可 grep 确认的事实。**

## Procedure

### Phase 1: 收集双方系统全量代码

**系统 A（通常是目标库）**：
1. `list_directory` 获取完整文件树
2. `read_file` 读取所有 .cs/.ts/.go 源文件
3. 读取项目文件（.csproj）确认依赖
4. 读取现有文档（README.md、设计文档）

**系统 B（通常是集成方）**：
1. 确定集成相关的关键文件（同步服务、日志、存储、配置）
2. `read_file` 逐文件读取所有相关代码
3. 确认共享基础设施（数据库连接工厂、日志接口、配置模型）

### Phase 2: 提取双方同步机制的事实

对每个关键维度，在双方源码中各搜索一次，记录确切的实现方式：

| 维度 | 搜索方法 | 示例 |
|------|----------|------|
| 增量同步键 | grep `_Synced\|_UploadFlag\|ProcessTime` 在查询条件中 | `WHERE _Synced = 0` vs `WHERE _UploadFlag > @time` |
| 水位线表 | grep `_SyncWatermark\|_SyncLog\|Watermark` 确认表名和结构 | `_SyncWatermark`（未使用）vs `_SyncLog`（正常运行） |
| 连接管理 | grep `new SqlSugarClient\|SharedLocalDb\|SqlSugarScope` | 各自创建 vs 共享单例 |
| 远程发现 | grep `sqlite_master\|TableNameList\|TableMappings` | 自动扫描 vs 配置白名单 |
| HTTP 调用 | grep `HttpClient\|_PostJson\|TryGetRemoteChunk` | 策略模式 vs 内建方法 |
| 重试策略 | grep `Retry\|退避\|backoff` | 指数退避 vs 无 |
| 日志接口 | grep `IStructuredLogger\|ILogger\|StructuredLogger` | 自定义接口 vs 内置实现 |
| 清理逻辑 | grep `Cleanup\|_CleanLogTimer\|Retention` | 按时间清理 vs 按水位线清理 |

### Phase 3: 差距分类与严重性评估

对每个发现的差异，按以下标准分类：

| 等级 | 标准 | 示例 |
|------|------|------|
| **P0** | 会导致集成直接失败 | 增量键不兼容 → 数据行无法查询；连接冲突 → SQLite 锁死 |
| **P1** | 需要额外适配代码，但不影响核心功能 | 水位线表结构不同；配置体系不统一 |
| **P2** | 可接受的差异，可通过配置或文档解决 | 表发现方式不同；批量大小硬编码差异 |

### Phase 4: 集成方案评估

对每种可能的集成路径，评估：

| 方案 | 评估维度 | 方法 |
|------|----------|------|
| 保留现状 | 工作量=0，风险=最低 | 短期可行，但失去新库优势 |
| 完全替换 | 工作量=最大，风险=最高 | 需要改造所有关键组件 |
| 增强后替换 | 工作量=中，风险=中 | **推荐**：识别最小改造集，逐步骤实现 |

### Phase 5: 输出集成差距文档

文档应包含：

1. **系统 A 当前架构** — 数据流图 + 关键源码事实（带文件行号引用）
2. **系统 B 当前架构** — 同上
3. **两套方案对比表** — 每个维度的具体实现对比
4. **差距分析** — 每个差距的问题描述、源码引用、影响分析、解决方案
5. **集成决策矩阵** — 各方案的工作量/风险/建议
6. **代码层面对比表** — 具体类/方法的实现差异
7. **集成验证清单** — 待确认的待办事项列表

## Verification Checklist

输出前逐项确认：

- [ ] 差距描述中的每个类名、方法名、字段名与双方源码一致
- [ ] 每个差距都有具体的源码行号引用
- [ ] 无"看起来应该"的推断 — 所有描述都有 grep/read_file 确认
- [ ] README.md 中的描述不与源码矛盾（如有矛盾，以源码为准并标注）
- [ ] 差距严重性等级有明确的判断标准支撑
- [ ] 集成方案的工作量估算基于实际修改文件数和修改点

## Anti-Hallucination Rules

1. **集成方可能没有你以为有的配置** — 例如 AppConfigRoot 中不一定有 DataSync 配置节
2. **集成方可能不用你以为的字段** — 例如数据行可能不含 `_Synced` 而是用 `_UploadFlag`
3. **接口名称相同 ≠ 语义相同** — 两个系统可能都有 `ILogger` 但签名不同
4. **"看起来兼容" ≠ 实际兼容** — 必须验证实际的数据格式、表结构、API 协议
5. **新库的 README 示例可能引用不存在的类型** — 以源码文件列表为准

## Key Patterns

### Pattern 1: 增量键不兼容

当两个系统使用不同的增量同步标记（`_Synced` vs `_UploadFlag`），直接导致一方无法被另一方识别。

**解决方案**：
- 方案 A: 新增兼容两种增量键的策略类（推荐，保持向后兼容）
- 方案 B: 在数据写入侧统一标记格式（需要改造集成方代码）

### Pattern 2: 连接冲突

两个组件各自创建数据库连接，导致 SQLite 并发锁死或 PostgreSQL 连接池耗尽。

**解决方案**：
- 修改组件 A 的构造函数，支持传入外部连接实例（可选参数，保持向后兼容）
- 或使用工厂模式 + DI 容器管理生命周期

### Pattern 3: 配置体系不一致

两个系统有不同的配置模型，需要映射或适配器。

**解决方案**：
- 新增配置适配层（在集成方侧新增配置节）
- 或统一配置模型，提供双向转换方法

### Pattern 4: 差距验证后发现已有部分基础设施

**经验教训**: 差距分析中的某些"差距"可能已在现有代码中以部分形式存在（如构造函数已接受共享连接参数）。开始编码前应验证现有基础设施，避免重复工作。

**流程**:
1. 差距分析文档 → 列出差距清单
2. 验证阶段 → grep 现有代码确认哪些差距已有部分实现
3. 改造阶段 → 只补全缺失部分，不重新实现已有部分
