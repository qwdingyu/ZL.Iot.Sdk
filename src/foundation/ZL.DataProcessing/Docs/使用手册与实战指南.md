# ZL.DataProcessing 使用手册与实战指南

## 1. 简介

**ZL.DataProcessing** 是一个专为 **工业自动化、仪器仪表模拟、协议测试** 领域打造的高性能数据处理引擎。它集成了**动态脚本执行、二进制/文本模式匹配、数据编解码、校验算法**等核心能力，旨在通过极其简洁的配置，解决复杂的通信协议模拟与数据处理难题。

---

## 2. 核心功能概览

| 功能模块                   | 描述                                                              | 关键词                       |
| :------------------------- | :---------------------------------------------------------------- | :--------------------------- |
| **DataExpressionEngine**   | 强大的脚本引擎，支持在字符串模板中嵌入 C# 表达式，实现动态逻辑。  | 脚本执行、数学运算、状态管理 |
| **PatternMatcher**         | 工业级模式匹配器，支持变长二进制匹配、十六进制通配符。            | `??`, `**`, `??[n-m]`        |
| **BinaryLogic / HexCodec** | 全面的数据转换工具，涵盖字节序、位操作、BCD 码、Hex 转换。        | 大小端、位掩码、数据清洗     |
| **ChecksumHelper**         | 常用校验算法计算（SUM/XOR/CRC8/CRC16/CRC32 等）。                | CRC16, CRC32, SUM, XOR       |
| **Context Management**     | 支持会话级（Session）与全局级（Global）状态隔离（多连接）。      | 会话状态、状态记忆           |

## 3. 语法速查手册

### 3.1 模板内嵌脚本

在响应模板中，你可以直接嵌入逻辑：

- **求值表达式 `${...}`**：计算结果并替换为字符串。
  - `Voltage: ${State['V'] * 1.5} V`
- **执行语句 `@{...}`**：仅执行逻辑（如修改状态），不产生输出。
  - `@{State['Power'] = 'ON'}`
- **全脚本模式**：如果模板以 `=` 开头，整段被视为脚本。
  - `=Hex.ToString(Crc.Modbus(Hex.ToBytes(Args[0])))`

### 3.2 二进制匹配通配符 (PatternMatcher)

用于判断接收到的 Hex 数据是否符合规则：

- `AA BB CC` : 精确匹配。
- `??` : 匹配任意 1 个字节。
- `??[3]` : 匹配任意 3 个字节。
- `??[2-5]` : 匹配 2 到 5 个字节（变长）。
- `**` : 匹配任意数量字节（0 到 无穷大）。

### 3.3 内置脚本函数库 (Helpers)

脚本中可直接使用的静态类：

- **Hex**: `Hex.ToBytes("AA BB")`, `Hex.FromInt(123, 2)`, `Hex.ToString(bytes, " ")`
- **Binary**: `Binary.GetBit(val, 0)`, `Binary.SwapBytes(bytes)`
- **Crc**: `Crc.Modbus(bytes)`, `Crc.Crc32(bytes)`
- **Checksum**: `Checksum.Calculate(bytes, "CRC16_X25")`, `Checksum.ToHex(bytes, "SUM16_LE", " ")`
- **Bit**: `Bit.Set(byteVal, 2, true)`
- **Format**: `Format.Binary(10, 8)`, `Format.Bcd(99)`
- **Random**: `Random.Next(0, 100)`
- **Args / ArgsText / Input**: `Args` 是捕获组数组，`ArgsText` 是捕获组拼接文本，`Input` 为原始输入。

**ChecksumHelper 支持的算法名称（部分）：**
`SUM8/SUM16/SUM32`、`XOR8/XOR16/XOR32`、`CRC8/CRC8_ITU/CRC8_MAXIM/CRC8_SAE_J1850/CRC8_ROHC`  
`CRC16/CRC16_MODBUS/CRC16_CCITT/CRC16_IBM/CRC16_X25/CRC16_XMODEM/CRC16_USB/CRC16_DNP/CRC16_KERMIT/CRC16_AUG_CCITT`  
`CRC32/CRC32C/CRC32_MPEG2`（以及 `_BE`/`_LE` 的 SUM/XOR 变体）。

## 4. 十大实战场景 (Use Cases)

以下场景涵盖了从简单的 SCPI 模拟到复杂的工业协议交互，充分展示类库威力。

### 场景一：模拟传感器数据的动态波动 (SCPI)

**需求：** 模拟一个温度传感器，每次查询时，返回一个在 25.0 左右微小波动的随机值，而不是死板的固定值。

- **输入命令：** `MEAS:TEMP?`
- **响应模板：**
  ```text
  ${ (25.0 + (Random.NextDouble() * 0.5 - 0.25)).ToString("F2") }
  ```
- **输出示例：** `25.12`, `24.98`, `25.05`

### 场景二：带状态记忆的继电器控制 (Stateful Logic)

**需求：** 发送“打开”命令后，查询状态应返回“ON”；发送“关闭”后，查询应返回“OFF”。

- **命令 1 (打开)：** `RELAY:ON`
  - **模板：** `@{State['Relay']='ON'}OK`
- **命令 2 (关闭)：** `RELAY:OFF`
  - **模板：** `@{State['Relay']='OFF'}OK`
- **命令 3 (查询)：** `RELAY:STAT?`
  - **模板：** `${State['Relay'] ?? 'OFF'}`

### 场景三：Modbus RTU 协议模拟 (自动计算 CRC)

**需求：** 模拟 Modbus 从站，返回数据时必须自动计算正确的 CRC16 校验码，否则主站会报错。
**假设：** 我们要返回地址 01，功能码 03，数据长度 02，数据内容 1234 (Hex)。

- **响应模板：**
  ```text
  @{
     var payload = Hex.ToBytes("01 03 02 12 34");
     var crc = Crc.Modbus(payload);
  }
  ${Hex.ToString(payload, " ")} ${Hex.ToString(crc, " ")}
  ```
- **输出结果：** `01 03 02 12 34 B5 33` (自动算好了校验)

### 场景四：变长数据包匹配与解析 (Binary Matching)

**需求：** 协议格式为 `Header(AA 55) + Len(1 byte) + Payload(N bytes) + Tail(FF)`。需要匹配这种变长帧并提取中间的 Payload。

- **匹配规则 (Pattern)：**
  ```text
  AA 55 ?? ** FF
  ```
  _(解释：Header 匹配，Len 匹配任意 1 字节，Payload 匹配任意长度，Tail 匹配 FF)_
- **处理脚本：** 假设通过正则或逻辑提取到了中间的 Payload，我们可以对其进行处理。

### 场景五：PLC 位状态模拟 (Bit Manipulation)

**需求：** 模拟一个 PLC 寄存器（1 个字节），其中第 0 位代表电机运行，第 2 位代表故障报警。需要单独修改某一位而不影响其他位。

- **初始状态：** `@{State['Reg'] = 0}`
- **设置故障命令：** `ALARM:SET`
  - **模板：**
  ```text
  @{
     byte current = Convert.ToByte(State['Reg']);
     State['Reg'] = Bit.Set(current, 2, true);
  }
  OK
  ```
- **读取寄存器命令：** `REG:READ?`
  - **模板：** `${Hex.FromInt(Convert.ToInt32(State['Reg']), 1)}`

### 场景六：大端/小端字节序自动转换 (Endianness)

**需求：** 模拟器内部使用整数存储数据，但通信协议要求以“小端模式”发送 4 字节整数。

- **当前值：** `State['Count'] = 100000` (Hex: 00 01 86 A0)
- **响应模板：**
  ```text
  HEADER ${Hex.FromInt(Convert.ToInt32(State['Count']), 4, true)} TAIL
  ```
- **输出结果：** `HEADER A0 86 01 00 TAIL` (自动转成了小端)

### 场景七：故障注入与模糊测试 (Fuzzing)

**需求：** 作为一个测试工具，故意发送错误的 CRC 校验码，或者故意延迟响应，来测试上位机的容错能力。

- **配置变量：** `State['InjectError'] = true`
- **响应模板：**
  ```text
  @{
     var data = Hex.ToBytes("01 03 00 00");
     var crc = Crc.Modbus(data);
     // 如果注入错误开启，人为修改 CRC
     if(Convert.ToBoolean(State['InjectError'])) {
         crc[0] = (byte)(crc[0] + 1);
     }
  }
  ${Hex.ToString(data, " ")} ${Hex.ToString(crc, " ")}
  ```

### 场景八：BCD 码时间格式模拟

**需求：** 许多电表使用 BCD 码传输时间。模拟读取当前时间，返回 `HH MM SS` 的 BCD 格式。

- **命令：** `READ:TIME`
- **响应模板：**
  ```text
  ${Format.Bcd(DateTime.Now.Hour)} ${Format.Bcd(DateTime.Now.Minute)} ${Format.Bcd(DateTime.Now.Second)}
  ```
- **输出示例：** `14 30 09` (代表 14:30:09)

### 场景九：请求参数回显与计算 (Echo & Math)

**需求：** 接收命令 `SET:VOLT 12.5`，响应需要确认设置值，并返回对应电流（假设阻抗 10 欧姆）。

- **正则匹配组 (Args)：** `Args[0]` 为 "12.5"
- **响应模板：**
  ```text
  OK, VOLT set to ${Args[0]}, Est. Current: ${ (Convert.ToDouble(Args[0]) / 10.0).ToString("F2") } A
  ```
- **输出结果：** `OK, VOLT set to 12.5, Est. Current: 1.25 A`

### 场景十：多步握手流程控制 (Session Flow)

**需求：** 模拟固件升级流程。第一步必须发 "HELLO"，第二步发 "AUTH"，否则第三步发 "DATA" 时报错。

- **命令 HELLO：** `@{Session['Step']=1} READY`
- **命令 AUTH：**
  ```text
  ${ if(Convert.ToInt32(Session['Step']) == 1) { Session['Step']=2; return "AUTH OK"; } else { return "ERR: SEQ"; } }
  ```
- **命令 DATA：**
  ```text
  ${ if(Convert.ToInt32(Session['Step']) == 2) { return "RECV OK"; } else { return "ERR: NO AUTH"; } }
  ```

### 场景十一：校验算法快速切换 (Checksum Switch)

**需求：** 上位机要求可配置校验算法，模板中根据配置切换。

- **配置：** `State['CRC'] = "CRC16_X25"`
- **响应模板：**
  ```text
  @{
     var data = Hex.ToBytes("AA 01 02 03");
     var crc = Checksum.ToHex(data, State["CRC"].ToString(), " ");
  }
  ${Hex.ToString(data, " ")} ${crc}
  ```

## 5. 性能与最佳实践

1.  **缓存优先**：`TemplateParser` 与表达式解析均带缓存机制。相同的模板/脚本会被复用，但缓存有上限，避免无限增长。高频通信建议保持模板稳定性。
2.  **错误处理**：脚本中建议做好类型转换的防御性编程。虽然引擎有 Try-Catch 兜底，但明确的 `Convert.ToInt32()` 比隐式转换更安全。
3.  **复杂逻辑外置**：如果逻辑极其复杂（如加密算法），建议通过 `DataExpressionEngine` 注册自定义的 C# 静态方法，而不是在模板里写几十行代码。

> 说明：多连接模式下，系统会为每个连接分配独立 Session。响应会按连接回送，避免状态串扰。
