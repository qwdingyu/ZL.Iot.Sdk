# ZL.ConnectionGuard 📚 工业级链路守护核心库

**“不再让连接断开成为现场事故。稳如泰山，守护每一字节。”**

## 1. 简介 (Introduction)

**ZL.ConnectionGuard** 是一个专为工业自动化（Industrial Automation）和物联网（IoT）场景设计的 .NET 通讯中间件。

它不负责具体的协议解析（如 Modbus 寄存器怎么读），而是专注于**“链路治理”**。它将物理连接（串口、TCP、CAN、PLC SDK）抽象为统一的**“卫士”**模型，彻底解决了工业现场最令人头疼的**断线重连、虚假连接、资源泄露、多线程死锁**等底层问题。

### 核心价值

- 🛡️ **工业级稳定性**：内置指数退避重连算法、看门狗超时检测、心跳保活机制。
- 🚀 **高性能**：底层采用 `ArrayPool` 内存池与 `SemaphoreSlim` 异步锁，零 GC 压力，高并发安全。
- 🔌 **万能适配器**：通过 `IChannelAdapter` 接口，抹平 RS232/485、TCP/IP、HSL、CAN 总线等所有硬件差异。
- 🧪 **仿真友好**：提供虚拟适配器，无需硬件即可开发上位机逻辑。

## 2. 快速入门 (Quick Start)

### 2.1 安装

引用 `ZL.ConnectionGuard.dll` 及相关依赖（如 `System.IO.Ports`）。如需 NLog，直接注入 `NLogLogger`。

### 2.2 极简示例：3 行代码守护一个串口

```csharp
// 1. 创建适配器 (Adapter) - 这里是串口，也可以是 TCP
var adapter = new SerialAdapter("COM3", 9600);

// 2. 创建卫士 (Guard) - 注入适配器
using var guard = new ConnectionGuard(adapter);

// 3. 启动！卫士会自动打开串口，并在断开时无限重试
guard.Start();

// 4. 订阅数据
guard.OnDataReceived += (data) => Console.WriteLine($"收到: {BitConverter.ToString(data)}");
```

## 3. 详细应用场景 (Usage Scenarios)

**ZL.ConnectionGuard** 的设计初衷是“一次编写，到处连接”。以下是典型场景：

### 场景 A：传统设备数据采集 (RS232/485)

**适用**：扫码枪、电子秤、温控器、旧式 PLC。
**痛点解决**：解决了 Windows 串口拔掉后 `Close()` 导致的线程死锁（Deadlock）问题。

```csharp
// 配置：COM1, 9600, N, 8, 1
var serialAdapter = new SerialAdapter("COM1", 9600);

var options = new ConnectionGuardOptions
{
    ReconnectMinDelayMs = 1000, // 掉线后，最快1秒重连
    DeviceDeadTimeoutMs = 5000  // 如果5秒没收到电子秤数据，认为设备假死，强制重启连接
};

var scannerGuard = new ConnectionGuard(serialAdapter, options);
scannerGuard.OnDataReceived += (data) => { /* 解析条码 */ };
scannerGuard.Start();
```

### 场景 B：网络设备与 Modbus TCP

**适用**：新型 PLC、远程 IO 模块、机械臂。
**痛点解决**：处理 TCP 粘包/半包（通过 Adapter），自动维护长连接心跳。

```csharp
// 连接西门子 S7-1200 (ModbusTCP 模式)
// Adapter 内部已处理 Modbus 帧头，OnDataReceived 吐出的直接是 UnitId + PDU
var mbAdapter = new ModbusTcpAdapter("192.168.1.10", 502, unitId: 1);

var plcGuard = new ConnectionGuard(mbAdapter, new ConnectionGuardOptions
{
    // 应用层心跳：每3秒发一次读取保持寄存器 0 的指令，确保链路存活
    HeartbeatIntervalMs = 3000,
    HeartbeatStrategy = new StaticHeartbeat(new byte[] { 0x03, 0x00, 0x00, 0x00, 0x01 }) // 功能码03, 起始0, 长度1
});

plcGuard.Start();
// 发送指令（线程安全，无需 lock）
await plcGuard.SendAsync(new byte[] { 0x06, 0x00, 0x0A, 0x00, 0x64 }); // 写寄存器10 = 100
```

### 场景 C：集成第三方 SDK (如 HSL / PLC 厂商库)

**适用**：使用 HslCommunication、Keyence SDK 等专用协议库。
**痛点解决**：第三方库的重连机制通常不可控或不统一。Guard 将其封装，统一管理生命周期。

```csharp
// 假设你有一个 HSL 的三菱对象
var melsec = new MelsecMcNet("192.168.1.20", 6000);

// HslAdapter 会反射调用 Connect/Close，不仅限于 HSL，任何 SDK 都能封装
var hslAdapter = new HslAdapter(melsec);

var guard = new ConnectionGuard(hslAdapter);
guard.OnStateChanged += (state, msg) =>
{
    if (state == GuardState.Connected)
        Console.WriteLine("三菱 PLC 已连接，准备就绪");
};
guard.Start();
```

### 场景 D：无硬件仿真开发 (Simulation)

**适用**：出差在高铁上写代码、演示给客户看、单元测试。
**痛点解决**：无需携带笨重的 PLC，代码逻辑零修改。

```csharp
// 仅需更换 Adapter，业务代码完全不动
IChannelAdapter adapter;

if (IsProduction)
    adapter = new TcpClientAdapter("192.168.1.5", 502);
else
    adapter = new VirtualAdapter(); // 虚拟适配器，会模拟回环数据

var guard = new ConnectionGuard(adapter);
guard.Start();
```

## 4. 高级配置 (Configuration)

通过 `ConnectionGuardOptions` 可以微调卫士的行为，以适应不同的工业现场要求。

| 属性                  | 默认值   | 说明                                                                 |
| :-------------------- | :------- | :------------------------------------------------------------------- |
| `ReconnectMinDelayMs` | 1000     | 首次重连等待时间（毫秒）。                                           |
| `ReconnectMaxDelayMs` | 30000    | 最大重连等待时间（指数退避上限）。                                   |
| `HeartbeatIntervalMs` | 3000     | 心跳包发送间隔。仅当 `HeartbeatStrategy` 或 `HeartbeatData` 设置时生效。 |
| `HeartbeatStrategy`   | null     | 心跳策略接口，用于动态生成心跳包（优先于 `HeartbeatData`）。           |
| `HeartbeatData`       | null     | 心跳包字节数组（已废弃）。若设置，空闲时会自动发送此数据保活。        |
| `DeviceDeadTimeoutMs` | 0 (禁用) | **看门狗超时**。若超过此时间未收到任何数据，视为连接假死，强制重连。 |
| `SendLockTimeoutMs`   | 2000     | 发送锁等待超时。防止多线程竞争导致 UI 卡死。                         |
| `MaintenanceLoopDelayMs` | 100     | 维护循环的检测周期。                                                |

## 5. 最佳实践 (Best Practices)

1.  **单例模式**：对于同一个物理设备，整个应用程序生命周期内应该只有一个 `ConnectionGuard` 实例。
2.  **UI 线程适配**：`OnDataReceived` 和 `OnStateChanged` 是在后台线程触发的。如果在 WinForm/WPF 更新 UI，请务必使用 `Invoke`。
    ```csharp
    guard.OnDataReceived += (data) => {
        this.Invoke(() => txtLog.AppendText(BitConverter.ToString(data)));
    };
    ```
3.  **日志注入**：在生产环境，务必注入 `NLog` 或 `Serilog` 封装的 `IGuardLogger`，以便事后分析断线原因（是网络波动还是驱动崩溃）。
4.  **不要在回调里做耗时操作**：`OnDataReceived` 应该只负责将数据推入队列或快速更新状态。复杂的数据库写入请另开线程。
5.  **心跳策略应保证稳定**：`HeartbeatStrategy` 内部不要抛异常，避免频繁错误日志；优先使用策略代替已废弃的 `HeartbeatData`。

## 6. 常见问题 (FAQ)

**Q: 为什么不用 HSL 自带的 `ConnectServer`？**
A: HSL 专注于协议解析（如把 byte[] 转成 float 温度）。`ConnectionGuard` 专注于连接的**存活**。两者结合使用效果最佳：Guard 负责把路修通，HSL 负责在路上跑车。

**Q: 支持 UDP/HTTP/CAN/LIN 吗？**
A: 支持。内置 `UdpClientAdapter` / `HttpPollingAdapter` / `CanAdapter` / `LinAdapter`。如需接入硬件 SDK，只需要实现对应的 Driver 接口即可。

**Q: 发送失败会抛异常吗？**
A: 不会。`SendAsync` 返回 `bool` (false 表示失败)。设计哲学是：**工业现场网络波动是常态，不要用异常流程来处理常态。** 失败后 Guard 会自动处理重连，你只需决定是否重发。
