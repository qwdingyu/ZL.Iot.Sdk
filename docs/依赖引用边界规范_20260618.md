# 依赖引用边界规范

> 版本：1.0
> 日期：2026-06-18
> 适用范围：`iot-sdk`、`ZL.PlcBase`、`UseThink.Iot/api`、`tmom` 等消费者项目

## 目标

这份规范解决三个问题：

1. 排查问题时能快速判断责任边界。
2. 消费者项目编译时不重编共享库源码。
3. Agent 和人工修改 `.csproj` 时有统一、可验证的规则。

## 核心规则

| 场景 | 必须使用 | 禁止使用 |
|---|---|---|
| `iot-sdk` 仓库内部项目互相引用 | `ProjectReference` | 指向 iot-sdk 自身包的 `PackageReference` |
| `ZL.PlcBase` 仓库内部项目互相引用 | `ProjectReference` | 指向 ZL.PlcBase 自身包的 `PackageReference` |
| `iot-sdk` 引用 `ZL.PlcBase` 能力 | `PackageReference` + `local-feed` | 跨仓库 `ProjectReference` |
| `UseThink.Iot/api` 引用 `iot-sdk` / `ZL.PlcBase` | `PackageReference` + `local-feed` | 跨仓库 `ProjectReference` |
| 第三方依赖 | `PackageReference` + CPM | `.csproj` 内写 `Version` |

一句话原则：

> 同仓库源码协作，跨仓库包契约交付。

## 为什么这样做

同仓库使用 `ProjectReference`：

- 修改后立即生效，不需要 pack / restore。
- 编译器看到同一份源码，类型不会漂移。
- 调试时能直接进入源码。

跨仓库使用 `PackageReference`：

- 消费者只编译自己的代码，编译时间更短。
- 问题边界清楚：包版本固定，先确认包本身还是消费者代码问题。
- `deploy-fast.sh` 可以把 `ZL.PlcBase` 和 `iot-sdk` 同步打包到 `~/.nuget/local-feed`，不需要发布到 nuget.org 才能联调。

## NuGet 源规则

本地开发机器的包源顺序必须是：

```xml
<packageSources>
  <clear />
  <add key="local-feed" value="/Users/dingyuwang/.nuget/local-feed" />
  <add key="nuget.org" value="https://api.nuget.org/v3/index.json" />
</packageSources>
```

禁止：

- 使用 `~/`，NuGet 不会展开这个路径。
- 使用 `$(MSBuildThisFileDirectory)` 等 MSBuild 属性。
- 把 `/Users/dingyuwang/0-X/iot-sdk/artifacts` 或 `/Users/dingyuwang/0-X/ZL.PlcBase/artifacts` 直接放进消费者 `NuGet.config`。

`artifacts` 是构建输出目录，不是稳定包源。所有构建产物必须统一进入：

```text
/Users/dingyuwang/.nuget/local-feed
```

## 标准联调流程

修改 `ZL.PlcBase` 或 `iot-sdk` 后：

```bash
/Users/dingyuwang/0-X/deploy/tools/deploy-fast.sh 2.2.2
```

然后在消费者项目中：

```bash
dotnet restore
dotnet build
```

如果复用同一个版本号覆盖本地包，必须清理相关 NuGet 缓存，否则消费者可能仍然使用旧包：

```bash
dotnet nuget locals http-cache -c
dotnet nuget locals global-packages -c
```

更推荐本地联调使用唯一版本号，例如：

```text
2.2.2-local.20260618.001
```

## Agent 强制执行规则

任何 agent 修改以下文件后，必须运行引用边界检查：

- `*.csproj`
- `Directory.Packages.props`
- `NuGet.config`
- `pipeline.json`

检查命令：

```bash
python3 scripts/verify-reference-boundaries.py
```

检查消费者项目：

```bash
python3 scripts/verify-reference-boundaries.py \
  --root /Users/dingyuwang/0-X/UseThink.Iot/api \
  --mode consumer
```

如果检查失败，不允许提交，不允许发布，不允许继续让消费者 restore。

## 允许的 ZL 外部包

`iot-sdk` 允许通过 NuGet 引用的 `ZL.*` 外部包只有：

| 包 | 来源仓库 | 原因 |
|---|---|---|
| `ZL.IotHub` | `ZL.PlcBase` | IoT 驱动核心 |
| `ZL.IotHub.Bridges` | `ZL.PlcBase` | 桥接能力 |
| `ZL.PFLite` | `ZL.PlcBase` | 通用基础工具库 |
| `ZL.Iot.Runner.Templates` | 历史模板包 | 仅允许生成器模板兼容场景使用，不允许新业务代码引用 |

禁止直接引用：

| 包 | 原因 |
|---|---|
| `ZL.Tag` | Tag 类型由 `ZL.IotHub` 承载，直接引用会重新引入 CS0433 风险 |

新增任何 `ZL.*` 外部包前，必须先更新本文档和 `scripts/verify-reference-boundaries.py` 的允许列表。

## 目录清理规则

必须删除或忽略：

- `bin/`
- `obj/`
- `artifacts/`
- `.dotnet-tools/`
- `.atomcode/`
- `.qwen/`

不能随意删除：

- `docs/inherited-from-tmom/`：这是历史迁移参考资料，不是构建产物。
- `docs/` 下的历史审计报告：除非确认已被新规范替代，并在新文档中说明迁移关系。
