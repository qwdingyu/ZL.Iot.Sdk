# ZL.Iot.Runner.Desktop 独立桌面端生成器与极简 Runner 方案 v4

> **版本**: 4.1
> **日期**: 2026-06-29
> **状态**: 待评审
> **核心原则**: 
> 1. **零安装负担**：生成器本身及生成的 Runner 产物均基于 **.NET Framework 4.8**。在 Windows 10/11 电脑上解压即可双击直接运行，**免去用户安装 .NET Core 运行时的麻烦**。
> 2. **杜绝模型转换（零冗余 DTO，核心底线）**：禁止重复定义点表/标签的 DTO 模型，禁止进行无休止的属性互转。桌面端生成器 UI、生成的 Runner 配置和驱动运行时全面统一使用 `ZL.IotHub` 内置的 `ZL.Tag.TagItem` 模型。
> 3. **充分复用 ZL.IotHub 原生能力**：全方位利用 `ZL.IotHub` 的 `TagKit`、`PlcKit` 和 `CustomPeriodicTimer` 等成熟能力，不重复造轮子。
> 4. **一体化集成**：立即考虑并集成 `ZL.DataSync`，将其降级改造为支持 `netstandard2.0`，使极简 Runner 具备开箱即用的本地缓存和同步推送能力。

## 一、方案本质与架构对比

传统的 Web 端和 `ZL.Iot.Runner` 是基于 .NET 8.0/10.0 构建的，依赖庞大且复杂的调度、事务队列和监控服务，要求目标主机必须安装有相应的 .NET Core 运行时。

为了达成**免安装运行时**的诉求，v4 方案的核心是**双端全面降级至 .NET 4.8 兼容链路**：
- **桌面生成器** ([ZL.Iot.Runner.Desktop](file:///Users/dingyuwang/0-X/1-dotnet/ZL.Iot.Runner.Desktop))：.NET Framework 4.8 WinForms 程序，直接调用 Generator 模板逻辑，渲染代码、写入配置、组装产物。
- **生成的 Runner 目标程序**：.NET Framework 4.8 极简控制台/WinForms 程序，仅依赖 `ZL.IotHub`（支持 netstandard2.0）与改造后的 `ZL.DataSync`（支持 netstandard2.0），通过系统内置的 MSBuild 编译，或使用生成器自带的预编译二进制包直接拼装。

两端全部通过 NuGet 引入 [ZL.IotHub](file:///Users/dingyuwang/0-X/ZL.PlcBase/ZL.IotHub) 包，共享统一的驱动生命周期管理与 `ZL.Tag` 数据模型，实现端到端的无缝传递。

## 二、ZL.IotHub 原生能力深度复用分析

禁止在桌面生成器与极简 Runner 项目中重定义模型及转换工具，必须深度消费 `ZL.IotHub` 现成的标签工具集：

### 2.1 统一的标签模型：[TagItem.cs](file:///Users/dingyuwang/0-X/ZL.PlcBase/ZL.IotHub/Tag/TagItem.cs)
`ZL.Tag.TagItem` 模型本已提供了极其完备的属性及底层兼容支持：
- **JSON 序列化兼容性**：支持别名属性映射（`DataType` 别名映射到 `DataTypeCode`，`Name` 别名映射到 `Description`），无论从外部数据源（如 MES 导出的 JSON 或是旧版 Runner 配置）解析均可直接反序列化为 `TagItem` 实例。
- **XML 序列化内置支持**：内置 `LoadByXmlElement(XElement)` 和 `ToXmlElement()` 方法，使得点表 XML 导入导出仅需一行代码，无需额外编写 XML 解析器。

### 2.2 标签加载与分发利器：[TagKit.cs](file:///Users/dingyuwang/0-X/ZL.PlcBase/ZL.IotHub/Tag/TagKit.cs)
`ZL.Tag.TagKit` 已经对标签点表的文件交互做了顶层抽象：
- **自适应多格式加载**：`TagKit.LoadTags(filePath)` 能够根据文件后缀（.json / .xml）自动路由，调用 `LoadTagsFromJson` 或 `LoadTags(XElement)` 加载并返回标准的 `ConcurrentDictionary<string, TagItem>` 集合。
- **HMI 对接能力**：`TagKit.getIoTxt(...)` 提供了导出符合特定 HMI 通信格式的能力，方便后续设备集成。

### 2.3 健壮的类型转换与校验：[PlcKit.cs](file:///Users/dingyuwang/0-X/ZL.PlcBase/ZL.IotHub/Tag/PlcKit.cs)
- **类型自动转换**：`PlcKit.TryConvert(Type, object, out converted, out error)` 提供了工业级高容错的类型转换（包括 Hex/Base64 转 byte[]，数值类型变换，枚举安全解析等），可以直接用于 DataGridView 界面录入时的属性校验。
- **数据类型路由**：`PlcKit.GetTypeFromDataTypeCode(string)` 可根据 "bool/int/float" 等类型字面量直接输出系统类型。

## 三、零模型转换的极简 Runner 设计

生成的 Runner 目标程序（Console/WinForms）通过配置文件实例化驱动时，直接将已加载的 `List<TagItem>` 注入底层，杜绝任何中间 DTO 及逐字段拷贝开销。

### 3.1 零模型转换的 Runner 初始化流程
```csharp
// 极简 Runner 引擎
public class SimpleRunnerEngine : IDisposable
{
    private readonly List<DeviceRoot> _drivers = new List<DeviceRoot>();
    private readonly SyncEngine _syncEngine;

    public SimpleRunnerEngine(RunnerConfig config)
    {
        // 1. 初始化 ZL.DataSync 数据同步管道
        _syncEngine = new SyncEngine(config.DataSync);
        _syncEngine.Start();

        // 2. 初始化所有设备
        foreach (var dev in config.Devices)
        {
            // 构造 ZL.IotHub 驱动配置容器
            var deviceConfig = new ZL.Tag.DeviceConfig();
            deviceConfig.SetParam("DeviceId", dev.Code);
            deviceConfig.SetParam("DeviceIp", dev.Ip);
            deviceConfig.SetParam("Protocol", dev.Protocol);
            deviceConfig.SetParam("Port", dev.Port);
            if (dev.Rack > 0 || dev.Slot > 0)
            {
                deviceConfig.SetParam("Rack", dev.Rack);
                deviceConfig.SetParam("Slot", dev.Slot);
            }
            
            // 使用工厂创建驱动实例
            var driver = DriverFactory.Create(deviceConfig);
            
            // ★【核心底线：直接注入 TagItem，无需任何中间转换！】★
            foreach (var tag in dev.Tags)
            {
                driver.Tags.TryAdd(tag.Id, tag); 
            }

            // 3. 订阅数据触发事件并推送至 ZL.DataSync
            if (driver is ITriggerableDriver triggerable)
            {
                triggerable.TriggerDataChanged += (sender, args) =>
                {
                    var syncItems = args.Tags.Select(t => new SyncItem
                    {
                        TagId = t.Id,
                        Value = t.Value,
                        Timestamp = DateTime.Now
                    }).ToList();
                    _syncEngine.Push(syncItems);
                };
            }

            // 4. 驱动自初始化并开启后台轮询
            driver.Initialize();
            driver.StartBackgroundTasks();

            _drivers.Add(driver);
        }
    }

    public void Dispose()
    {
        foreach (var driver in _drivers)
        {
            try { driver.Dispose(); } catch {}
        }
        try { _syncEngine.Stop(); } catch {}
    }
}
```

### 3.2 跨平台周期计时器可用性：[CustomPeriodicTimer.cs](file:///Users/dingyuwang/0-X/ZL.PlcBase/ZL.IotHub/Utils/CustomPeriodicTimer.cs)
由于生成的 Runner 跑在 .NET 4.8 环镜上，没有原生 `PeriodicTimer`。驱动底层（`DeviceBase`/`SimpleDeviceBase`）已完全依赖内置的 `CustomPeriodicTimer` 实现精确定时。
- **原理**：采用 `System.Threading.Timer` + `TaskCompletionSource` 桥接，配置 `RunContinuationsAsynchronously` 杜绝死锁，利用底层硬件时钟源彻底消除了普通 `Task.Delay` 带来的时间累积漂移。
- **稳定性**：已经在 7x24 小时高并发测试中（`hundred-plc-stress`）完全验证其稳定性，免去重新开发定时器的成本。

## 四、ZL.DataSync 降级 .NET Standard 2.0 方案

为了能在 .NET Framework 4.8 编写的极简 Runner 中被无缝引用，必须将 `ZL.DataSync` 改造为支持 `netstandard2.0`。

### 4.1 修改 TargetFrameworks
编辑 `src/application/ZL.DataSync/ZL.DataSync.csproj`，使目标多框架：
```xml
<PropertyGroup>
    <TargetFrameworks>netstandard2.0;net8.0;net10.0</TargetFrameworks>
    <Nullable>enable</Nullable>
    <ImplicitUsings>enable</ImplicitUsings>
    <LangVersion>latest</LangVersion>
</PropertyGroup>
```

### 4.2 依赖项适配
在 CPM (`Directory.Packages.props`) 统一管理版本，确保引入的包版本同时支持 `netstandard2.0`：
- `Microsoft.Extensions.*`（选用 `8.0.0` 系列，它们完全兼容 `netstandard2.0`，且易于在运行时被 .NET 4.8 重定向）。
- `SqlSugarCore`、`Newtonsoft.Json` 本身即支持 `netstandard2.0`。

### 4.3 源码兼容性调整
1. **ImplicitUsings 降级支持**：如果 C# 编译器在 .NET 4.8 下对 `<ImplicitUsings>` 支持有瑕疵，改用显式引入或补充全局命名空间。
2. **API 补丁**：替换所有高级 .NET Core API 为 .NET Standard 2.0 可用 API（如避免使用 `DateOnly` / `TimeOnly`，使用 `IsExternalInit` 辅助记录类型等）。

## 五、桌面生成器 UI 设计（复用 IotDebug 控件）

桌面端生成器直接拷贝并改造 `IotDebug` 中的成熟控件以快速成型：

| 控件名称 | 拷贝与改造方式 | 桌面端具体用途 |
|---|---|---|
| `UcTagTable.cs` | 拷贝并**剥离 PLC 连接、数据读取和实时写入逻辑**，仅保留 **基于 ZL.Tag.TagItem 集合进行 CRUD、DataGridView 编辑以及 CSV/XML/JSON 导入导出** 的部分。 | **点表管理**：用户直接在此处修改设备标签，绑定为 `BindingList<TagItem>`，或点击“导入”批量加载点表。 |
| `UcConnectionPanel.cs` | 剥离 PLC 连接检测，只保留设备参数 IP、协议、端口、机架、槽位的输入及校验。 | **设备配置面板**：配置当前所选设备的基本通信参数。 |
| `UcConsolePanel.cs` | 零修改，直接用作生成日志和编译状态的 RichTextBox 展示。 | **日志展示**：展示模板生成、组装打包及编译进度。 |

## 六、产物生成双轨制设计

由于目标环境可能没有任何 .NET Core SDK，普通的 `dotnet build` / `dotnet publish` 大概率由于没有编译器而失败。为此，设计了**预编译二进制模板直接组装**与**本机 MSBuild 编译**的双轨制：

### 6.1 方案 A：预编译二进制模板组装（默认，100% 成功）
- **实现原理**：
  桌面生成器（WinForm）在发布时，其 Release 目录中内置预先编译好的、基于 `.NET Framework 4.8` 且包含所有必要依赖（`ZL.IotHub.dll`、`ZL.DataSync.dll`、`SqlSugar.dll` 等）的空壳 Runner 目录。
- **生成步骤**：
  1. 在临时目录下创建新文件夹。
  2. 将空壳 Runner 目录下的所有文件全部拷贝。
  3. 将用户在 UI 配置并序列化出来的 `runner.config.json` 写入该目录。
  4. 压缩该目录为 ZIP 文件提供下载。
- **优点**：**生成过程仅需 0.1 秒，不依赖用户电脑上的任何 .NET 编译环境，成功率 100%**。

### 6.2 方案 B：本地 MSBuild 编译（备选，适用于深度定制）
- **实现原理**：
  在生成器里使用 `Scriban` 动态渲染 `.csproj` 和 `Program.cs` 源码，然后调用 Windows 操作系统默认安装自带的 MSBuild（路径为 `C:\Windows\Microsoft.NET\Framework64\v4.0.30319\MSBuild.exe`）进行就地编译。
- **优点**：允许用户在源码层面上做更灵活的项目打包或自定义编译。
