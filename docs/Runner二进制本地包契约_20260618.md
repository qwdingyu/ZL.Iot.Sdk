# Runner 二进制本地包契约

> 日期：2026-06-18  
> 范围：`ZL.Iot.Runner.Generator` Binary 模式、`ZL.Iot.Runner` 运行时配置校验、local-feed 发布链路。

## 背景

UseThink.Iot 通过 local-feed 消费 iot-sdk 包。Runner Generator 的 Binary 模式会在临时目录生成 Console 项目并执行 `dotnet restore/publish`，该临时目录不属于 iot-sdk 仓库，也不会天然继承仓库级 `NuGet.config`。

如果 Binary 模式只依赖当前进程所在目录的隐式 NuGet 配置，会出现以下问题：

- 临时项目找不到 `/Users/dingyuwang/.nuget/local-feed` 中的 `ZL.*` 包。
- 本机全局 NuGet 缓存存在旧包时，可能恢复到旧版本，造成运行时程序集版本不一致。
- self-contained RID 发布首次下载 runtime pack 时，60 秒 restore 超时过短，容易误判失败。
- `AssemblyVersion` 与 NuGet 包版本不是同一个概念。Runner 程序集可能仍是 `1.0.0`，但包版本是 `2.2.1`。生成器不能用程序集版本反推 PackageReference 版本。
- IoT 驱动链路包含反射和运行时加载，单文件发布容易遗漏 `ZL.IotHub`、HSL、OPC 等动态依赖。

## 规则

1. Binary Generator 在临时项目目录中必须存在 `NuGet.config`。
2. 优先复制调用方当前目录向上查找到的最近 `NuGet.config`。
3. 找不到时写入最小配置：`local-feed` + `nuget.org`。
4. iot-sdk 内部仍遵守仓库内 `ProjectReference`、跨仓库 `PackageReference + local-feed` 的引用边界。
5. Runner 配置校验必须同时接受 legacy 协议名和标准协议 key，例如 `ModbusTcp` 与 `modbus-tcp`。
6. 生成项目中的 `ZL.Iot.Runner` 与 `ZL.IotHub` 必须使用同一个 NuGet 包版本。版本解析顺序：
   - 环境变量 `ZL_IOT_RUNNER_PACKAGE_VERSION`。
   - 当前进程 `.deps.json` 中的 `ZL.Iot.Runner.Generator` / `ZL.Iot.Runner` 包版本。
   - Generator 程序集 informational version。
   - 最后才退回 `1.0.0`。
7. Binary 模式默认采用 self-contained **目录发布**，不再生成单文件。目录发布是 Runner 的标准交付形态，ZIP 中必须包含 exe/dll、配置文件、日志配置和 manifest。

## 验证

UseThink.Iot 的真实二进制 smoke 已覆盖：

```bash
cd /Users/dingyuwang/0-X/UseThink.Iot/api
./scripts/smoke-runner-binary.sh
```

通过标准：

- 启动 `PlcSimulator.Cli` 的 Modbus TCP 端口。
- 生成 Console Binary ZIP。
- 解压并启动 Runner。
- `runner.config.json` 被 Runner 正常读取。
- SQLite `iot_tag_history` 表出现新增记录。
- `manifest.json` 与 `runner.config.json` 基础字段校验通过。
- ZIP 中包含 `ZL.IotHub.dll` 等驱动依赖，不依赖目标机全局缓存。

## 注意

`ZL.DataSync` 作为 Runner 的运行时依赖包，必须按当前发布目标生成对应 TFM 的程序集。当前包同时面向 `net8.0` 和 `net10.0`，避免 `net10.0` Runner 发布时恢复到缺失程序集的包。

协议 key 以 `ZL.IotHub` 注册表为准。新增协议时先进入 `ZL.IotHub` 的 `NativeProtocolRegistry` / `HslProtocolRegistry`，Runner 只消费注册结果，不再在 `ConfigLoader` 中维护另一份手写白名单。
