# iot-sdk 五层架构迁移方案

## 目标

将 iot-sdk 的 18 个项目从顶层平铺改为五层架构。**完全按照 tmom/CLAUDE.md 规范**：
- **RTK**: 所有 shell 命令前缀 `rtk`
- **原子提交**: 每完成一个文件修改立即 `rtk git add` + `rtk git commit`
- **NuGet.config 铁律**: 不使用 `$(MSBuildThisFileDirectory)`，改用相对路径或默认

## 层级依赖规则（强制）

```
foundation  →  platform  →  domain  →  application  →  app
   ↑                                                    │
   └────────────────────────────────────────────────────┘
   允许：低层 → 高层单向引用；不允许反向引用
```

**特例**（允许）：
- `domain → platform → foundation` （向下）
- `application → domain → platform → foundation` （向下）
- `tests → 任意层`（测试可以引用任何层，但只能放在 tests/）
- 任何层 → `ZL.Shared`（在 foundation）

## 实施步骤（按原子提交原则）

### 步骤 1：创建分层目录 + 用 `cp -R` 批量迁移
- 用 `cp -R` 拷贝 5 个业务库到 `iot-sdk/src/platform/`
- 用 `cp -R` 拷贝 3 个 DAO + 1 个 Biz 到 `iot-sdk/src/domain/`
- 用 `cp -R` 拷贝 5 个 service/app 到 `iot-sdk/src/application/`
- 用 `cp -R` 拷贝 2 个 Cli 到 `iot-sdk/src/app/`
- 用 `cp -R` 拷贝 4 个 Tests 到 `iot-sdk/tests/`

### 步骤 2：修正所有 ProjectReference 路径
- 相对路径从 `..\ZL.X\ZL.X.csproj` 改为 `..\..\X\ZL.X.csproj`（跨层）或保持同层 `..\ZL.X\ZL.X.csproj`

### 步骤 3：Directory.Packages.props 补充缺失版本
- 补 `System.IO.Ports`, `DynamicExpresso.Core`, `Serilog.*`

### 步骤 4：pipeline.json 自动发现
- 改为通配 `src/*/ZL.*/ZL.*.csproj`（排除 tests 和 cli）

### 步骤 5：修正 ZL.Simulator 引用路径
- 改用 NuGet 引用（先发布 foundation 5 个库到 NuGet，本地项目再用 NuGet）

### 步骤 6：执行发布（先 dry-run → 实际发布）
- 用 `rtk zl-pipeline publish <ver>` 一键发布

## 风险与对策

| 风险 | 对策 |
|------|------|
| 路径错位导致编译失败 | 改完后用 `rtk dotnet restore` + `rtk dotnet build` 立即验证 |
| ProjectReference 跨层路径错误 | 路径从 `..\ZL.X\ZL.X.csproj` 改 `..\..\platform\ZL.X\ZL.X.csproj` |
| NuGet 引用 vs 本地引用混淆 | 顶层 iot-sdk 项目间使用 ProjectReference；foundation 5 个库通过 NuGet 引用 |
| pipeline.json 路径过期 | 改用通配 `glob` 模式 |
| git mv 丢历史 | 用 `rtk git mv` 而不是 `rm` + `cp`，保持文件历史 |

## 验收标准

1. ✅ `rtk dotnet build iot-sdk/IoT.Sdk.sln` 0 错误
2. ✅ `bash iot-sdk/scripts/verify-nuget-config.sh` 全通过
3. ✅ `rtk zl-pipeline publish 1.0.0 --dry-run` 全部 8 步通过
4. ✅ NuGet 上 5 个 foundation 包已发布 v1.0.0
5. ✅ ZL.Simulator 编译通过，引用新的 NuGet 包（不再本地引用）
