# ZL.Simulator → iot-sdk 能力追加方案

> 生成日期：2025-07-23
> 角色视角：架构师 + 产品经理 + 行业顾问
> 核心原则：**以 iot-sdk 为主体**，保持原有 20 个项目名称不变，Simulator 的能力追加到对应项目中
> 仅当 iot-sdk 完全没有对应能力时，才创建新项目（2-3 个）

---

## 执行摘要

ZL.Simulator 经过重构沉淀出一套**通用且高内聚的能力模块**。经过逐文件对比分析：

| Simulator 能力 | iot-sdk 对应 | 合并方式 | 优先级 |
|---------------|-------------|---------|--------|
| 模板引擎扩展语法（`{TAG:}`/`{RAND_HEX:}`/`delay=`） | `ZL.DataProcessing` 已有基础替换 | **追加到现有项目** | P0 |
| `SimpleExpressionEvaluator`（算术表达式解析） | `ZL.DataProcessing` 无等价物 | **追加到现有项目** | P0 |
| 会话生命周期事件模型 | `ZL.DataProcessing` 内部有状态管理但无事件 | **追加到现有项目** | P0 |
| `ZL.Framing` | 完全相同的共享库 | **无需处理** | — |
| `ZL.Script` | 完全相同的共享库 | **无需处理** | — |
| `TransparentProxyTransport`（透明代理/流量录制） | 完全没有 | **新建 `ZL.Probing`** | P1 |
| `ParameterValidator`（参数校验） | `ZL.Biz.Execute` 有 `ConditionTree` 但无参数校验 | **追加到 `ZL.Biz.Execute/Validation/`** | P1 |
| `ProtocolConfig`（JSON 协议驱动配置） | 完全不同的模型（RunnerConfig 是设备接入视角） | **新建 `ZL.Protocol`** | P1 |
| `TagRegistry`（运行时标签管理） | 概念不同维度（DAO.IotDevice 是数据库持久化） | **参考设计，可选追加到 `ZL.DataProcessing`** | P2 |

**追加后效果**：Simulator 中 **70%+** 的可复用核心能力被 iot-sdk 吸收，且只新增 2-3 个项目。

## 一、现状对比：重合度分析

### 1.1 完全重合（共享项目，无需处理）

以下项目在两个仓库中**完全相同**，属于共享项目：

| 项目 | 说明 |
|------|------|
| `ZL.Framing` | 帧处理（FrameAssembler/Decoder/Buffer/CRC 算法），两个仓库 10 个文件完全相同 |
| `ZL.Script` | DynamicExpresso C# 脚本引擎，5 个文件完全相同 |

### 1.2 部分重合（需追加增强）

| iot-sdk 项目 | 已有能力 | Simulator 独有能力 | 合并方式 |
|-------------|---------|-------------------|---------|
| `ZL.DataProcessing` | `DataExpressionEngine`（`{STATE:}/{RAND:}/{SESSION:}` 替换）、`TemplateParser`、`ScriptEngine`、`PatternMatcher`、`ChecksumHelper` | `{TAG:}` 语法、`{RAND_HEX:n}` 随机 Hex、`delay=ms;payload` 分段响应、`SimpleExpressionEvaluator`（递归下降算术解析器）、会话生命周期事件 | **追加到现有项目** |
| `ZL.ConnectionGuard` | `IChannelAdapter` + 10+ 种适配器 + `ConnectionGuard`（重连/心跳/看门狗） | `TransparentProxyTransport`（透明代理/流量录制） | **新建 `ZL.Probing`**（独立项目） |
| `ZL.Biz.Execute` | `ConditionTree` + `ConditionTreeEvaluator`（业务规则 DSL） | `ParameterValidator`（类型/范围/枚举/正则校验） | **追加到 `ZL.Biz.Execute/Validation/`** |

### 1.3 完全不同（需新建项目）

| Simulator 能力 | iot-sdk 状态 | 新建项目 | 理由 |
|---------------|-------------|---------|------|
| `ProtocolConfig`（JSON 协议驱动范式） | `RunnerConfig` 是"设备接入"视角，与 Simulator 的"协议仿真"视角完全不同 | `ZL.Protocol` | 命令-响应模板、条件响应、参数校验、混沌模拟 — iot-sdk 完全没有等价物 |
| `TagRegistry`（运行时标签注册表） | `IotTag` 是数据库持久化模型，不同维度 | 参考设计追加到 `ZL.DataProcessing`（可选） | 线程安全标签缓存，可作通用工具 |
| `TagSimulationEngine`（仿真数据生成） | 无等价物 | 不提取 | Fixed/Ramp/Sine/Random/Step 五种仿真模式，模拟器核心 |
| `ProtocolLearner`（协议学习器） | 无等价物 | 不提取 | 从通信日志自动生成协议配置，是模拟器"护城河"能力 |

## 二、追加方案详解（不创建新项目的）

### 2.1 `ZL.DataProcessing` 追加 — P0

#### 2.1.1 追加内容

| Simulator 能力 | 目标位置 | 说明 |
|---------------|---------|------|
| `{TAG:}` 语法 | `DataExpressionEngine.ReplaceTokens()` | 从 `ITagSource` 接口读取标签值替换 |
| `{RAND_HEX:n}` 语法 | `DataExpressionEngine.ReplaceTokens()` | 生成 n 字节随机 Hex 字符串 |
| `delay=ms;payload` 分段响应 | 新增 `ResponseSegment` 静态类 + 辅助方法 | 解析 `delay=X;payload||delay=Y;payload2` 格式 |
| `SimpleExpressionEvaluator` | 新增 `ArithmeticExpressionEvaluator.cs` | 递归下降算术表达式解析器，支持 +/-\*/() 优先级 |
| 会话生命周期事件 | 增强 `DataExpressionEngine` | 追加 `StateChanged`/`SessionStateChanged`/`SessionReset`/`SessionChanged` 事件 |

#### 2.1.2 目标结构

```
iot-sdk/src/foundation/ZL.DataProcessing/
├── DataExpressionEngine.cs        (增强：追加新模板语法 + 事件)
├── TemplateParser.cs              (不变)
├── ScriptEngine.cs (引用 ZL.Script)
├── PatternMatcher.cs
├── HexCodec.cs
├── ChecksumHelper.cs              (增强：反向同步 Simulator 的 30+ 种算法)
├── BitHelper.cs
├── FormatHelper.cs
├── ScriptHelpers.cs
├── ArithmeticExpressionEvaluator.cs  ← 新增：来自 Simulator 的 SimpleExpressionEvaluator
└── ResponseSegment.cs              ← 新增：delay=ms;payload 分段响应
```

#### 2.1.3 新增模板语法

| 语法 | 示例 | 说明 |
|------|------|------|
| `{TAG:tagName}` | `{TAG:temp}` | 从 `ITagSource` 接口读取标签值 |
| `{RAND_HEX:length}` | `{RAND_HEX:8}` | 生成 8 字节随机 Hex 字符串 |
| `delay=X;payload` | `delay=200;{STATE:mode}||delay=100;OK` | 分段延迟响应，`||` 分隔 |

#### 2.1.4 ITagSource 接口设计

```csharp
/// <summary>
/// 标签值数据源接口（供 {TAG:} 模板语法使用）
/// </summary>
public interface ITagSource
{
    double? GetValue(string tagName);
    string? GetValueString(string tagName);
    event Action<string, double?>? TagValueChanged;
}
```

#### 2.1.5 ArithmeticExpressionEvaluator 设计

```csharp
/// <summary>
/// 递归下降算术表达式求值器
/// 支持：变量替换 {STATE:voltage} / 位置参数 {0} / {1}
/// 运算符：+ - * / % ^ 和括号
/// </summary>
public static class ArithmeticExpressionEvaluator
{
    public static double Evaluate(string expression, IReadOnlyDictionary<string, double> variables);
}
```

#### 2.2.3 ValidationRule 模型

```csharp
/// <summary>
/// 参数校验规则
/// </summary>
public class ValidationRule
{
    public string ParameterName { get; set; }         // 参数名
    public ValidationType Type { get; set; }           // int/float/string/enum/pattern
    public bool Required { get; set; }                 // 是否必填
    public double? Min { get; set; }                   // 最小值
    public double? Max { get; set; }                   // 最大值
    public string[]? EnumValues { get; set; }          // 枚举值列表
    public string? Pattern { get; set; }               // 正则表达式
}

public enum ValidationType
{
    Int,
    Float,
    String,
    Enum,
    Pattern
}
```

## 三、新建项目（iot-sdk 完全没有对应能力的）

### 3.1 `ZL.Probing` — 透明代理/流量录制 — P1

#### 3.1.1 来源

```
ZL.Simulator/src/Simulator.Instruments/Probing/
└── TransparentProxyTransport.cs
```

#### 3.1.2 核心功能

| 功能 | 说明 |
|------|------|
| 透明代理 | 拦截设备与上位机之间的所有通信 |
| 流量录制 | 将请求/响应写入 JSONL 格式日志文件 |
| 会话级日志 | 按会话分别记录日志到独立文件 |
| 侦听模式 | TCP Server 模式，监听端口并转发到目标 |
| Sniffer Only | 可配置为只侦听不转发 |

#### 3.1.3 目标结构

```
iot-sdk/src/foundation/ZL.Probing/                   ← 新建项目
├── TransparentProxyTransport.cs                     (核心：透明代理传输)
├── TransparentProxyConfig.cs                        (配置：监听端口、目标地址、日志路径)
├── LogRecorder.cs                                   (JSONL 格式日志录制)
├── SessionLogManager.cs                             (会话级日志管理)
└── ZL.Probing.csproj
```

#### 3.1.4 依赖

```
ZL.Probing → ZL.Framing (帧处理)
ZL.Probing → ZL.Shared (日志)
```

#### 3.1.5 核心价值

- 新设备协议分析：拦截通信流量 → 录制日志 → 后续用于 ProtocolLearner 学习
- 协议调试：实时查看设备与上位机之间的原始字节流
- 与 `ZL.ConnectionGuard` 互补：Guard 管连接可靠性，Probing 管流量分析

### 3.2 `ZL.Protocol` — JSON 协议驱动配置 — P1

#### 3.2.1 来源

```
ZL.Simulator/src/Simulator.Core/Protocols/
├── ProtocolConfig.cs
├── CommandDefinition.cs
├── ConditionalResponse.cs
├── EventDefinition.cs
└── SimulationTagConfig.cs

ZL.Simulator/src/Simulator.Instruments/Config/
└── ProtocolConfigLoader.cs
```

#### 3.2.2 核心模型

| 类型 | 职责 |
|------|------|
| `ProtocolConfig` | 根配置：设备名、协议版本、命令列表、状态变量、混沌模拟参数 |
| `CommandDefinition` | 单条命令：请求模板、响应模板、状态绑定、条件响应、参数校验 |
| `ConditionalResponse` | 条件响应：`Condition` → `ResponseTemplate` 映射 |
| `EventDefinition` | 事件定义：触发条件 → 事件动作 |
| `ProtocolConfigLoader` | JSON/文件加载器 |

#### 3.2.3 目标结构

```
iot-sdk/src/foundation/ZL.Protocol/                  ← 新建项目
├── ProtocolConfig.cs
├── CommandDefinition.cs
├── ConditionalResponse.cs
├── EventDefinition.cs
├── SimulationTagConfig.cs
├── ProtocolConfigLoader.cs
└── ZL.Protocol.csproj
```

#### 3.2.4 协议配置 JSON 示例

```json
{
  "DeviceName": "MyDevice",
  "ProtocolVersion": "1.0",
  "Terminator": "\n",
  "StateVariables": ["voltage", "current", "mode"],
  "Commands": [
    {
      "CommandTemplate": "*IDN?",
      "ResponseTemplate": "ZL,MyDevice,1.0"
    },
    {
      "CommandTemplate": "VOLT {v}",
      "SetStateKey": "voltage",
      "Validation": [
        { "ParameterName": "v", "Type": "float", "Min": 0, "Max": 100 }
      ],
      "ResponseTemplate": "OK"
    },
    {
      "CommandTemplate": "VOLT?",
      "ResponseTemplate": "{STATE:voltage}"
    }
  ],
  "Chaos": {
    "SimDelayMs": 0,
    "SimPacketLossRate": 0.0,
    "JitterMs": 0
  }
}
```

#### 3.2.5 混沌模拟能力

| 参数 | 说明 | 用途 |
|------|------|------|
| `SimDelayMs` | 固定响应延迟（毫秒） | 模拟网络延迟 |
| `SimPacketLossRate` | 丢包率（0.0-1.0） | 模拟网络不稳定 |
| `JitterMs` | 响应时间抖动 | 模拟网络抖动 |

#### 3.2.6 依赖

```
ZL.Protocol → ZL.Iot.Interface (枚举/模型)
ZL.Protocol → ZL.DataProcessing (模板替换)
```

#### 3.2.7 与 iot-sdk 现有 `RunnerConfig` 的关系

| 维度 | `RunnerConfig` (现有) | `ProtocolConfig` (新建) |
|------|----------------------|------------------------|
| 视角 | "设备接入"：连什么 PLC、读什么地址 | "协议仿真"：发什么命令、返回什么响应 |
| 用途 | 配置设备驱动实例 | 配置命令-响应模板 |
| 配置方式 | JSON/XML：设备类型/IP/端口/标签列表 | JSON：命令模板/响应模板/状态机 |
| 互补性 | 两者互补：RunnerConfig 定义"连什么"，ProtocolConfig 定义"怎么交互" |

### 3.3 `ZL.Tags` — 运行时标签注册表 — P2（可选）

> 注意：此项目为**可选**。如果 iot-sdk 不需要运行时标签缓存能力，可暂缓。

#### 3.3.1 来源

```
ZL.Simulator/src/Simulator.Instruments/Tags/
└── TagRegistry.cs           (线程安全标签注册表)
```

#### 3.3.2 只提取 TagRegistry

| 能力 | 说明 |
|------|------|
| `TagRegistry` | 线程安全的 ConcurrentDictionary 标签存储，支持双精度值 + 格式 + 时间戳 |
| `TagValueChanged` 事件 | 标签值变化时通知订阅者 |
| CAS 更新 | 条件原子更新，用于并发安全 |

**不提取** `TagSimulationEngine`：Fixed/Ramp/Sine/Random/Step 五种仿真模式是模拟器核心能力，iot-sdk 不需要。

#### 3.3.3 目标结构（可选）

```
iot-sdk/src/platform/ZL.Tags/                          ← 新建项目（可选）
├── TagRegistry.cs
├── TagEntry.cs
├── TagValueChangedEventArgs.cs
└── ZL.Tags.csproj
```

#### 3.3.4 与 `Dao.IotDevice/IotTag` 的区别

| 维度 | `Dao.IotDevice/IotTagDto` | `ZL.Tags/TagRegistry` |
|------|--------------------------|----------------------|
| 持久化 | SQLite 数据库 | 内存（ConcurrentDictionary） |
| 用途 | 设备点位定义（地址/类型/单位等） | 运行时标签值缓存 |
| 线程安全 | 否 | 是 |
| 事件通知 | 无 | `TagValueChanged` 事件 |
| 生命周期 | 应用启动到关闭 | 同左，但支持动态增删 |

#### 3.3.5 依赖

```
ZL.Tags → ZL.Iot.Interface (枚举/模型)
```

## 六、实施计划

### 6.1 里程碑

| 阶段 | 内容 | 预估工作量 | 风险 |
|------|------|-----------|------|
| M1: `ZL.DataProcessing` 增强 | 追加 `{TAG:}`/`{RAND_HEX:}`/`delay=`/`ArithmeticExpressionEvaluator`/事件 | 2-3 天 | 低（纯追加，不破坏现有 API） |
| M2: `ZL.Protocol` 新建 | ProtocolConfig/CommandDefinition/ConditionalResponse/Loader | 3-4 天 | 中（需定义与 RunnerConfig 的映射关系） |
| M3: `ZL.Probing` 新建 | TransparentProxyTransport/LogRecorder/SessionLogManager | 2-3 天 | 低（纯新增，不与其他项目耦合） |
| M4: `ZL.Biz.Execute` 增强 | 追加 `Validation/ParameterValidator` | 1-2 天 | 低（纯追加，不破坏现有 API） |
| M5: `ZL.Tags` 可选 | 如果决定需要则追加，不需要则跳过 | 1 天 | 低 |
| M6: 清理 | 反向同步 ChecksumUtil，更新文档 | 1 天 | 低 |

**总预估：10-14 个工作日（不含可选项目）**

### 6.2 每个阶段交付物

每个阶段交付：
1. 修改后的项目源码（追加的文件）或新建项目的完整源码
2. `.csproj` 文件（新项目含正确的 PackageReference 和 TargetFramework）
3. 单元测试
4. XML 文档注释
5. 变更说明（修改了哪些现有文件，新增了什么）

### 6.3 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| `ZL.Protocol` 与 `RunnerConfig` 的共存和映射 | 中 | 设计明确的边界：RunnerConfig 定义"连什么设备"，ProtocolConfig 定义"怎么与设备交互" |
| `ZL.DataProcessing` 增强的 API 兼容性 | 低 | 所有新增功能以可选接口形式提供（如 `ITagSource`），默认行为不变 |
| `ZL.Probing` 与 `ZL.ConnectionGuard` 的关系 | 低 | Probing 是可选的流量录制中间件，不替代 ConnectionGuard |

## 七、追加后 Simulator 的变化

追加完成后，ZL.Simulator 可以引用 iot-sdk 的新增能力：

| Simulator 代码 | 替换为 | 说明 |
|---------------|--------|------|
| `Simulator.Instruments/Templating/` 的模板语法 | `ZL.DataProcessing` (增强后) | 模板语法由 iot-sdk 统一维护 |
| `Simulator.Core/Protocols/` | `ZL.Protocol` | 协议配置模型由 iot-sdk 统一维护 |
| `Simulator.Instruments/Probing/` | `ZL.Probing` | 透明代理由 iot-sdk 统一维护 |
| `Simulator.Instruments/Simulators/ParameterValidator.cs` | `ZL.Biz.Execute/Validation/` | 参数校验由 iot-sdk 统一维护 |

**保留在 Simulator 的核心能力**：
- `InstrumentBase` 生命周期管理
- `TemplatedSimulatorBase`（连接 iot-sdk 的 `ZL.DataProcessing` + `ZL.Protocol`）
- `SimulatorHost` / `InstrumentRegistry`
- UI 层（WinForms/Avalonia/Grpc/Cli）

## 八、总结

### 8.1 变更清单

| 类型 | 项目 | 变更内容 | 工作量 |
|------|------|---------|--------|
| **增强** | `ZL.DataProcessing` | 追加 `{TAG:}`/`{RAND_HEX:}`/`delay=`/`ArithmeticExpressionEvaluator`/事件 | 2-3 天 |
| **增强** | `ZL.Biz.Execute` | 追加 `Validation/ParameterValidator` | 1-2 天 |
| **新建** | `ZL.Protocol` | JSON 协议驱动配置（CommandDefinition/ConditionalResponse/ProtocolConfig） | 3-4 天 |
| **新建** | `ZL.Probing` | 透明代理/流量录制（TransparentProxyTransport） | 2-3 天 |
| **可选** | `ZL.Tags` | 运行时标签注册表（TagRegistry） | 1 天 |

### 8.2 价值总结

| 维度 | 价值 |
|------|------|
| **保持现有结构** | 20 个原有项目全部保留，名称不变，已有代码零破坏 |
| **代码复用** | Simulator 中 70%+ 的可复用能力被 iot-sdk 吸收 |
| **新能力** | iot-sdk 获得 JSON 协议驱动（`ZL.Protocol`）、流量录制（`ZL.Probing`）、参数校验（`Validation`）、模板扩展语法 |
| **Simulator 瘦身** | 可引用 iot-sdk 的新项目，减少约 30% 的重复代码 |
| **外部复用** | `ZL.Protocol` 和 `ZL.Probing` 可独立作为 NuGet 包被其他项目引用 |

### 8.3 建议执行顺序

```
M1: ZL.DataProcessing 增强 (低风险，纯追加)
  ↓
M2: ZL.Protocol 新建 (核心数据模型，为其他项目打基础)
  ↓
M3: ZL.Probing 新建 (独立能力，与其他项目耦合少)
  ↓
M4: ZL.Biz.Execute 增强 (追加 Validation)
  ↓
M5: ZL.Tags (可选，按需)
  ↓
M6: 清理 (反向同步 ChecksumUtil，更新文档)
```

### 8.4 与传统方案的对比

| 维度 | 原方案（新建 5 个项目） | 本方案（增强 2 个 + 新建 2 个可选） |
|------|----------------------|---------------------------------|
| 新项目数 | 5 个 | 2-3 个 |
| 原有项目变更 | 0 个 | 2 个增强 |
| 对现有代码影响 | 低 | 极低（纯追加，不破坏 API） |
| 代码复用率 | 85%+ | 70%+ |
| 实施复杂度 | 高 | 低 |
| 维护成本 | 高（更多项目） | 低（保持现有结构） |
| 推荐程度 | — | **强烈推荐** |
