### 1.3 具体问题源码分析

#### S1: verify 命令缺少 version

```python
# L428-432: 当前代码
def cmd_verify(args):
    args.dry_run = True       # ✅ 设置了 dry_run
    cmd_publish(args)          # ❌ args.version 不存在！
```

`cmd_publish` L180: `version = args.version` → `AttributeError: 'Namespace'= object' has no attribute 'version'`

**修复方案：**

```python
def cmd_verify(args):
    args.dry_run = True
    # verify 需要版本来进行 pack 验证
    if not hasattr(args, 'version') or not args.version:
        print("[ERROR] verify 需要版本号")
        print("[HINT]  运行: zl-pipeline verify 1.0.0")
        sys.exit(1)
    cmd_publish(args)
```

同时 argparse 中 verify 需要添加 version 参数：

```python
verify_parser.add_argument("version", help="版本号, e.g. 1.0.0")
```

#### S2: 无超时保护的 subprocess 调用

全部构建命令使用 `capture_output=True` 但无 `timeout`：

```python
# L208-211: 典型示例
result = run(
    ["dotnet", "build", str(csproj), "-c", "Release", "--nologo", "-v", "q"],
    capture_output=True, text=True, dry_run=dry_run  # ❌ 无 timeout
)
```

当 MSBuild 节点卡死、NuGet 恢复挂起、或 Obfuscar 进入死循环时，**整个流水线无限等待**。

**修复方案：** 在 `run()` 函数中注入默认超时：

```python
DEFAULT_TIMEOUTS = {
    "dotnet_build": 300,    # 5 分钟
    "dotnet_pack": 180,     # 3 分钟
    "dotnet_publish": 300,  # 5 分钟
    "dotnet_nuget_push": 120,
    "obfuscar": 300,
    "python_script": 60,
    "bash_script": 120,
    "dotnet_version": 10,
    "which": 5,
}

def run(cmd, timeout=None, **kwargs):
    if timeout is None:
        cmd0 = cmd[0] if isinstance(cmd, list) else cmd
        if "dotnet" in cmd0:
            timeout = DEFAULT_TIMEOUTS.get("dotnet_build", 300)
        elif "obfuscar" in cmd0:
            timeout = DEFAULT_TIMEOUTS["obfuscar"]
        elif sys.executable in (cmd0, str(cmd0)):
            timeout = DEFAULT_TIMEOUTS["python_script"]
        else:
            timeout = 60
    kwargs["timeout"] = timeout
    try:
        return subprocess.run(cmd, **kwargs)
    except subprocess.TimeoutExpired:
        print(f"  [TIMEOUT] {' '.join(cmd[:3])}... 超过 {timeout}s")
        raise
```

#### M1: MSBuild NodeReuse 泄漏

```python
# 在 main() 开头添加
import os
os.environ["MSBUILDDISABLENODEREUSE"] = "1"
os.environ["PYTHONUNBUFFERED"] = "1"
```

### 1.4 流水线增强建议

| 增强项 | 说明 | 优先级 |
|--------|------|--------|
| **打包后 SHA256 校验** | pack 完成后计算 SHA256，push 完成后二次校验 | P2 |
| **版本自动递增** | `zl-pipeline publish` 不带版本号时自动读取当前最高 NuGet 版本并递增补丁号 | P2 |
| **失败后自动清理** | 构建失败后自动 `dotnet build-server shutdown` 清理残留 MSBuild 进程 | P1 |
| **并行 pack** | pack 阶段各包无依赖关系，可并行执行加速 | P3 |
| **结构化日志** | 输出 JSON 格式日志供 CI/CD 解析 | P3 |

### 3.2 关键约束分析

#### 约束 1: SqlSugarCore 需要 .NET Standard 2.1

```
SqlSugarCore 5.1.4.146 → 仅支持 netstandard2.1+
netstandard2.1 ≠ netstandard2.0
→ .NET Framework 4.8 不支持 netstandard2.1
→ 任何引用 SqlSugarCore 的项目无法支持 net48
```

**受影响项目：** ZL.DB.Acc（直接引用 SqlSugarCore）
**传递受影响：** ZL.Dao.IotDevice → ZL.DB.Acc, ZL.Dao.Edge → ZL.DB.Acc, ZL.Biz.Execute → ZL.Dao.IotDevice

#### 约束 2: Fleck 是纯 .NET Framework 库

```
Fleck 1.2.0 → 仅支持 .NET Framework 4.5+
→ 不支持 netstandard2.0
→ 仅在 net48 目标中可用
```

**受影响项目：** 直接引用 Fleck 的项目需条件引用

#### 约束 3: Microsoft.Data.Sqlite 8.0.1 仅支持 net8.0

```
Microsoft.Data.Sqlite 8.0.1 → 仅 net8.0
→ 不支持 netstandard2.0
→ 但若通过 SqlSugarCore 引用，SqlSugarCore 自带此依赖
```

## 4. 多目标框架方案设计

### 4.1 目标

| 目标 | 说明 |
|------|------|
| **net8.0** | LTS，当前主力，必须支持 |
| **net10.0** | 最新版本，向前兼容 |
| **netstandard2.0** | 兼容 .NET Framework 4.6.1+、.NET Core 2.0+、 Xamarin 等 |
| **net48** | 仅 A 组轻量库支持，满足遗留 Windows 场景 |

### 4.2 实施方案

#### 方案：Conditional Multi-Targeting

不在每个 csproj 中写 `<TargetFrameworks>`，而是通过 `Directory.Build.props` 条件注入：

```xml
<!-- Directory.Build.props -->
<Project>
  <!-- 统一元数据保持不变 -->
  <PropertyGroup>
    <Company>UseThink</Company>
    <Authors>UseThink IoT Team</Authors>
    <!-- ... 其他元数据 ... -->
    <LangVersion>latest</LangVersion>
    <Nullable>enable</Nullable>
    <ImplicitUsings>enable</ImplicitUsings>
  </PropertyGroup>

  <!--
    多目标框架策略：
    - A 组（纯工具库）：netstandard2.0;net48;net8.0;net10.0
    - B 组（数据库层）：net8.0;net10.0（SqlSugarCore 需要 netstandard2.1）
    - 默认：net8.0;net10.0
  -->

  <!-- A 组：轻量纯工具库 — 全平台支持 -->
  <PropertyGroup Condition="
    $(MSBuildProjectName) == 'ZL.Framing' Or
    $(MSBuildProjectName) == 'ZL.Script' Or
    $(MSBuildProjectName) == 'ZL.Scripting' Or
    $(MSBuildProjectName) == 'ZL.ConnectionGuard' Or
    $(MSBuildProjectName) == 'ZL.DataProcessing' Or
    $(MSBuildProjectName) == 'ZL.DataConvert' Or
    $(MSBuildProjectName) == 'ZL.Iot.Interface'">
    <TargetFrameworks>netstandard2.0;net48;net8.0;net10.0</TargetFrameworks>
  </PropertyGroup>

  <!-- A-Shared：日志库 — 不包含 net48（Serilog 4.x 不支持 net48 的全部功能） -->
  <PropertyGroup Condition="$(MSBuildProjectName) == 'ZL.Shared'">
    <TargetFrameworks>netstandard2.0;net8.0;net10.0</TargetFrameworks>
  </PropertyGroup>

  <!-- B 组：数据库及上层 — 仅 net8.0+（SqlSugarCore 约束） -->
  <PropertyGroup Condition="
    $(MSBuildProjectName) == 'ZL.DB.Acc' Or
    $(MSBuildProjectName) == 'ZL.Dao.IotDevice' Or
    $(MSBuildProjectName) == 'ZL.Dao.Edge' Or
    $(MSBuildProjectName) == 'ZL.Biz.Execute' Or
    $(MSBuildProjectName) == 'ZL.Iot.Plugin' Or
    $(MSBuildProjectName) == 'ZL.Iot.Runner' Or
    $(MSBuildProjectName) == 'ZL.Iot.Runner.Generator' Or
    $(MSBuildProjectName) == 'ZL.EdgeService'">
    <TargetFrameworks>net8.0;net10.0</TargetFrameworks>
  </PropertyGroup>

  <!-- CLI 应用 — 单目标 net8.0 -->
  <PropertyGroup Condition="$(MSBuildProjectName.EndsWith('Cli'))">
    <TargetFramework>net8.0</TargetFramework>
    <OutputType>Exe</OutputType>
  </PropertyGroup>

  <!-- 测试项目 — 单目标 net8.0 -->
  <PropertyGroup Condition="$(MSBuildProjectName.EndsWith('.Tests'))">
    <TargetFramework>net8.0</TargetFramework>
    <IsPackable>false</IsPackable>
  </PropertyGroup>

  <!-- netstandard2.0 特殊处理 -->
  <ItemGroup Condition="$(TargetFramework) == 'netstandard2.0'">
    <!-- nullable 在 netstandard2.0 中可能需要额外包 -->
    <PackageReference Include="System.Threading.Channels" Condition="!Exists('...')" />
  </ItemGroup>

  <!-- net48 特殊处理 -->
  <ItemGroup Condition="$(TargetFramework) == 'net48'">
    <PackageReference Include="System.Text.Json" VersionOverride="8.0.5" />
  </ItemGroup>

  <!-- 测试和 CLI 不打包 -->
  <PropertyGroup Condition="$(MSBuildProjectName.EndsWith('.Tests')) OR $(MSBuildProjectName.EndsWith('Cli'))">
    <IsPackable>false</IsPackable>
  </PropertyGroup>

  <!-- 源码链接 -->
  <ItemGroup>
    <PackageReference Include="Microsoft.SourceLink.GitHub" PrivateAssets="All" />
  </ItemGroup>
</Project>
```

### 6.1 历史采坑（本次审计确认）

| # | 坑 | 症状 | 根因 | 永久方案 |
|---|----|------|------|---------|
| P1 | **subprocess capture_output 无限阻塞** | 流水线卡死，无输出 | `capture_output=True` 在无 timeout 时，如果子进程不退出则永远等待 | `run()` 函数强制注入 timeout，所有调用不可绕过 |
| P2 | **MSBuild Node 资源泄漏** | 后续构建变慢或失败 | MSBuild 16.8+ 的 NodeReuse 特性，异常退出后节点不释放 | 脚本开头 `os.environ["MSBUILDDISABLENODEREUSE"] = "1"` |
| P3 | **NuGet CDN 传播延迟** | 刚发布的包 A，项目 B 引用 A 时 restore 失败 | NuGet.org CDN 全球同步需 1-5 分钟 | 发布间添加 `time.sleep(10)` 等待；或清除本地缓存重试 |
| P4 | **Obfuscar 破坏公共 API** | 消费者引用混淆后的包，编译报错找不到类型/方法 | Obfuscar 的 `KeepPublicApi=true` 不够，某些反射调用仍被破坏 | 纯反射库（ZL.Scripting, Generator）禁用混淆 |
| P5 | **System.Text.Json 版本冲突** | Scriban 7.1.0 要求 STJ >= 8.0.4，但项目引用 8.0.0 | 传递依赖版本不一致 | Central Package Management 统一提升到 9.0.14 |
| P6 | **NU5026 包内容不完整** | pack 警告 `was not found` | csproj 中 TargetFrameworks 指定了 netstandard2.0 但代码编译仅输出 net8.0 | csproj 中的 TargetFramework(s) 必须与实际编译输出一致 |
| P7 | **verify 命令崩溃** | `zl-pipeline verify` → AttributeError | verify subparser 未定义 version 参数，cmd_verify 未设置 | argparse 中添加 version，cmd_verify 中传递 |
| P8 | **dry_run 逻辑混乱** | do_push 中 dry_run 变量被反复赋值 | 多次 `dry_run = False; dry_run = ...or...` 冗余赋值 | 简化为单次计算 |

## 附录 A: 行业最佳实践参考

### A.1 NuGet 多目标框架策略

| 库类型 | 推荐 TFM | 理由 |
|--------|---------|------|
| 纯工具库（无平台依赖） | `netstandard2.0;net8.0` | 最大覆盖率，netstandard2.0 覆盖 .NET Framework 4.6.1+ 到 .NET 7 |
| 需要 .NET Framework 4.8 原生支持 | `netstandard2.0;net48;net8.0` | net48 提供 Framework 特有 API（如 Registry、COM） |
| 依赖现代 .NET 特性 | `net8.0` | LTS，稳定，覆盖最广的现代 .NET 场景 |
| 向前看 | `net8.0;net10.0` | 同时支持 LTS 和最新，但需确认所有依赖兼容 |

### A.2 CI/CD 自动化最佳实践

1. **Pipeline 脚本是单一事实来源** — 不要分散逻辑到 shell、Makefile、GitHub Actions 多处
2. **超时是一切子进程的底线** — 没有任何理由让构建命令无限等待
3. **Dry-run 必须模拟真实路径** — 不能跳过文件存在性检查
4. **版本由脚本管理** — 不要手动改版本号，用 `zl-pipeline publish --bump patch`
5. **发布前必须 API 对比** — 混淆/重构后确认公共 API 未丢失
6. **幂等性** — publish 命令可重复执行，`--skip-duplicate` 是必须的

### A.3 GitHub + NuGet 自动化架构

```
开发者本地
  └─ zl-pipeline publish 1.0.1    ← 手动控制版本，完整验证链

GitHub Actions (推荐后续接入)
  ├─ push tag v1.0.1
  ├─ dotnet restore + build (多框架)
  ├─ dotnet test (多框架)
  ├─ zl-pipeline publish (自动从 git tag 提取版本)
  └─ git tag + release notes

NuGet.org
  └─ 自动审核（开源包通常几分钟）
```
